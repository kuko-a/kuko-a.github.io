初学git

写在前面

为了把这个博客托管在github上，我已经了解了一些相应的git用法，github只不过是git的一部分，git适用于目前大部分的工作场景。git官网上也有着相应的教程，通篇的英文让人难以下手。好在国人写的教程也相当出彩，我这里觉得廖雪峰和菜鸟教程都对初学者很友好，前者引导较好而后者更为直观，互相对照着进行学习是一个好方法。

环境配置

git config命令可以配置或读取相应的工作环境变量，它决定了git在各个环节的工作方式与行为。
在使用git时需要先进行

git config --global user.name "kuko"
git config --global usre.email kuko2975245568@gamil.com

这两行命令配置了个人的用户名称和电子邮箱地址，--global选项定义了计算机上所有的git仓库，去掉的话则会把新个人信息保存在当前项目的.git/config文件之中。
除去必须定义的用户信息外，默认的文本编辑器和差异分析工具均可以进行二次配置，配置完成后可以使用git config --list来检查已有的配置信息。

基本逻辑概念

git通过工作区，暂存区，版本库，远程仓库来实现对数据的管理控制。

工作区

在电脑中能直接看到的目录成为工作区，用于在本地进行所需要的代码开发。这些代码还未纳入到被记录的版本控制中。

暂存库

暂存库是.git下的临时存储区域，保存了已经add过的文件，这些文件在下一次commit时会被全部提交至版本库中。
git和其他版本控制系统的区别是git跟踪并管理是修改并非文件，每次的add是修改本身，也就是说每次的add相当于一次快照。

版本库

在工作区下有着一个隐藏的目录.git，它就是git的版本库，里面主要包含了index、master、指向master的指针head。每次的提交commit都会在版本库中创建新的快照，他提供了项目完整历史记录的各个节点。

远程仓库

用24小时开机的机器充当服务器的角色，其他人可以不断从这个仓库中克隆新版本并且向他提交自己的推送，在github上可以创建一个git仓库来充当远程仓库。

基本操作

  • 创建一个版本库，git init把相应目录变成git可以实现管理的仓库。git add 可以将文件逐个添加到暂存区，之后使用git commit -m "注释"将文件全部提交到版本库。
  • 查看库中状态，git status可以查看什么文件发生了变化，git diff 文件可以给出文件具体改动的内容。
  • 版本回溯，git log可以显示从最近到最远的提交日志，想要回退到某个版本就得用git reset 参数 版本号，参数中的–hard会回退到已提交状态，–soft会回退到未提交状态，–mixed会回退到已添加但未提交的状态。在git中HEAD指向当前版本HEAD^是上个版本以此类推，如果^太多可以使用HEAD~100的形式。git reflog可以打开命令历史，从而找到回退之前的commit id，这样就能reset回去了。实际上commit id不用写完，写前几位就可以自动索引到。
  • 撤销修改，git checkout --文件可以实现文件在工作区的修改撤销，如果修改存在于add之前会撤销到和版本库一样的状态，修改存在于add之后就会回到add之后的状态。git reset HEAD 文件可以将暂存区的修改回退到工作区，此处的HEAD代表最新的版本。
  • 删除文件，rm 文件的作用和把文件放入回收站的效果一样，这时工作区里就没有了文件，git rm 文件可以删除版本库中的文件，再次commit更新删除效果。
  • 忽略文件，编写一个.gitgnore文件放在git工作区的根目录下，git将会自动忽略这些文件

Github相关

  • 关联github上的远程仓库，git remote add origin git@github.com：GitHub账户名/仓库名.git通常这个命令在github建立远程仓库时自己生成的，origin是默认的远程库的名字。
  • 推送到github上的远程库，git push -u origin master首次的push后面的-u参数使得本地master和远程master相关联，后续可以直接使用。
  • 删除远程库，git remote rm 仓库名可以解除本地和远程库的关联。git remote -v用来查看远程库的信息。
  • 克隆远程库，git clone repsitory-url通过复制的网址即可把远程库中的文件克隆到本地目录。

分支管理

git通过创建不同的分支来实现不同项目的各自开发，各分支可以进行合并。

  • 创建分支，git branch 分支名能新增一个指针默认名dev，git checkout dev会将HEAD的指向切换到dev上，之后工作区的修改和提交都是针对dev分支，master指针会留着原地不变。
  • 查看当前分支，git branch会列出所有分支，并在当前分支前标记*号。
  • 合并分支，git merge 分支名会将指定的分支合并到当前的分支并不留下痕迹git merge --no-ff -m"注释" 分支名会禁用fast forward，在分支历史上可以看出分支信息。
  • 部分合并，git cherry-pick 版本号可以复制这个版本号下的特定提交到当前的分支。
  • 删除分支，git branch -d 分支名可以删除指定的分支，一般在合并完后使用。git branch -D 分支名强行删除未合并的分支。
  • 冲突处理，当不同的分支处理了相同的文件时，合并分支会出现错误，此时需要查看冲突的文件并在工作区实现手动处理修正。
  • 储存当前环境，git stash会存储当前的工作区，git stash list查看储存的内容，git stash apply 文件名将储存的内容恢复到工作区，git stash drop用来删除stash的内容，git stash pop实现恢复和删除。

协作问题

合作开发时，所有人会往master和dev分支上推送各自的修改，当相同的文件受到不同的修改时，后者的推送会出现错误。

  • 获取最新版本，优先使用 git pull把最新的版本抓取下来，也就是git fetch和git merge的简化代码。抓取失败时需要指定本地dev分支与远程origin/dev分支的链接。
  • 合并和推送，将最新版本与本地版本合并后推送，如果有冲突就需要进行冲突处理再向上推送。
  • 变基，git base会实现分叉的提交历史整理成一条直线，多人的协作往往会使得提交历史乱七八糟。

标签管理

tag的作用是和commit相绑定，毕竟乱七八糟的数字不好找。

  • 创建标签，git tag 名称就可以在此分支上打一个新标签。git tag 名称 版本号可以对指定的commit id新打一个tag。
  • 查看标签，git tag可以查看所有的标签。
  • 推送标签，git push origin --tags推送所有标签给远程库。
  • 删除标签，git tag -d 名称能删除在本地的标签。删除完本地标签后，git push origin :refs/tags/名称会删除远程库中的标签。

一些感想

git是芬兰人Linus的杰作，实际上Linus仅仅用了两周时间完成了这个分布式版本控制系统的全部开发，可见Linus对于解决问题的执着和对技术的狂热。然而在我学习git的途中，最为佩服的其实并不是Linus无与伦比的编程技术，而佩服他基于技术自由和开放合作的精神。Linus最成功、最壮大的项目都是开源项目，这种强调自由共享和协作精神同共产主义的共享理念有着些许相通之处，这或许与他曾当选芬兰共产党中央委员会委员的父亲Nils Torvalds有关，但是Linus不是一位政治家，他只不过是在他擅长的领域去做他认为正确的事。明辨是非是所有成年人必须学会的一项技能。正如仁慈主义开展各种慈善晚会只是勉强遮掩社会贫困的真实根源一样，有很多事情需要自己的眼睛去看、自己的大脑去想，也没有必要上升到信仰的高度，仅仅做到言行合一，问心无愧已经算是成功。
